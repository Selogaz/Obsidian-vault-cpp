Тип проекта и команда
	- Что за продукт? Состав команды?
		- Проект - веб-платформа Verme, предназначенная для повышения эффективности управления линейным персоналом: учёт рабочего времени, автопланирование смен, биометрическая идентификация, инструменты для работы с внешним персоналом. Команда: 1 техлид, 2 бэкенда, 2 фронтенд-разработчика, 2 QA-инженера, аналитик, формализующий требования и составляющий ТЗ. Работали по двухнедельным спринтам в рамках Agile-подхода.
- Твои зоны ответственности
	- За какие модули/сервисы отвечал?
		- Реализовывал и поддерживал ключевые модули, связанные с управлением сменами и взаимодействием пользователей платформы - создание, редактирование и удаление смен, а также логику работы с комментариями и уведомлениями (например, модерация, уведомления об изменениях).
		- Использовал Spring Data JPA для описания связей между сущностями (сотрудник - смена - комментарий). Частично участвовал в развитии сервисов управления пользователями: настройка ролей, аутентификация и авторизация.
- Какие задачи решал регулярно?
	- Оптимизировал SQL-запросы к PostgreSQL для ускорения работы веб-интерфейса
	- Интегрировал backend-сервисы с фронтендом через REST API и с базой данных
	- Поддерживал корректную реализацию бизнес-логики, обрабатывал ошибки и крайние случаи
	- Писал юнит- и интеграционные тесты для покрытия критически важных сценариев
- Ключевые задачи
	- Какие фичи/модули лично реализовал?
		- Внедрил возможность сохранения черновиков смен, что было очень важно для HR-менеджеров
		- Реализовал базовую аутентификацию и авторизацию через Spring Security с разграничением ролей: администраторы, HR-менеджеры, линейные сотрудники - у каждого свой уровень доступа
		- Разработал REST API с использованием Spring MVC, который используется фронтендом и мобильными клиентами
	- Что оптимизировал/рефакторил?
		- Ускорил загрузку страниц с расписаниями и комментариями, оптимизировав Hibernate-запросы: заменил eager-загрузку на lazy, добавил выборочные JOIN, убрал N+1-проблемы. Рефакторил код модулей, улучшая структуру сервисов и выделяя общие части в отдельные утилитарные классы для повышения читаемости и поддержки.
	- С какими трудностями сталкивался и как решал?
		- Впервые работал с Spring Security - использовал ИИ для быстрого старта, изучал документацию, разбирал кейсы. Настроил фильтры безопасности, интеграцию с БД и защиту от несанкционированного доступа
- Расскажи о самой интересной/самой сложной задаче (самое важное из этого блока)
	- Кейс 1. Ролевой доступ в системе учёта смен
		- Нужно было четко разграничить действия пользователей: администраторы могут управлять всем, HR - только своими сменами, а линейный персонал - только просматривать и комментировать. Было тяжело быстро освоить Spring Security, настроить фильтры, обработку JWT-токенов и синхронизацию с внутренней моделью пользователя. Но в результате моя архитектура авторизации была включена в основной релиз.
	- Кейс 2: Производительность загрузки смен с комментариями
		- Проблема была следующая: страницы с расписаниями и длинными цепочками комментариев грузились медленно. Нужно было разобраться, почему запросы к базе работают медленно, и сократить время загрузки. Решение было следующим: выявил избыточные join'ы и излишнюю eager загрузку связанных сущностей. Переписал запросы, добавил lazy loading для комментариев, использовал выборочные join’ы только там, где это необходимо. Также ограничил глубину вложенности комментариев на странице и внедрил пагинацию. Результат - время отклика сократилось на 25%
- Технологии и инструменты
	- Что из стека использовал ?
		- Java 17, Spring Boot, Hibernate, PostgreSQL, Spring Security, REST API, Spring MVC, Maven, JUnit, Mockito, Docker.
	- Какие инструменты автоматизации/CI/CD?
		- Jenkins выполнял задачи по сборке проекта, запуску unit и интеграционных тестов, а также по сборке Docker-образов. GitLab CI/CD использовался для автоматизации деплоя на тестовые и продакшн окружения
- Тесты и качество
	- Как тестировал код?
		- Писал unit-тесты для бизнес-логики и сервисных слоев, создавая заглушки и имитации зависимостей. Интеграционные тесты запускал с использованием Spring Boot Test, проверяя взаимодействие компонентов и корректность работы с базой данных PostgreSQL в тестовом окружении.
	- Делал ли ревью чужого кода?
		- Нет, ревью кода не проводил. Буду рад получить такой опыт в вашей компании
- Результаты (здесь важно добавить цифры)
	- Что улучшилось благодаря твоим решениям? (метрики, скорость, стабильность)
		- Выделил переиспользуемые компоненты, уменьшил дублирование и обеспечил более чистое разделение логики. Благодаря этому, в будущем повысится скорость разработки новых фич. Сложно оценить насколько конкретно.
		- Переписал сложные запросы с использованием выборочных join-операций, что позволило сократить время выполнения операций выборки данных примерно на 25%
		- Внедрил модуль ролевого доступа
- Почему решил поменять проект?
	- Решил сменить проект, чтобы получить более разнообразный и глубокий опыт в backend-разработке, повысить свою ценность как разработчика. Мне интересно вносить вклад в компанию и расти вместе с продуктом.
