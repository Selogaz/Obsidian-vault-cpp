### Как работает кеш в проекте

Как работает кэш в проекте
Проект использует трёхуровневую систему кэширования для данных и ассетов визарда персонажа.
1. Модульный синглтон-промис (данные)
В src/lib/characterAssets.ts:94-114 — самый верхний уровень. Промис загрузки справочных данных (расы, классы, происхождения, мировоззрения) сохраняется в замыкании:
```js
let dataPromise: Promise<CharacterBootstrapData> | null = null;

export function loadCharacterData(): Promise<CharacterBootstrapData> {
	if (!dataPromise) {
		dataPromise = Promise.all([getRaces(), getClasses(), getBackgrounds(), getAlignments()])
			.then(([races, classes, backgrounds, alignments]) => ({ races, classes, backgrounds, alignments }));
	}
	return dataPromise;
}
```
Первый вызов фетчит с бэка, все последующие — возвращают тот же промис. Это бессрочный кэш на время жизни страницы (сброс только через resetCharacterCache()).
2. In-memory SVG-кэш
В src/lib/svgCache.ts — простая `Map<string, string>:`
`const svgCache = new Map<string, string>();`
`const inflight = new Map<string, Promise<void>>();`
- При первом запросе SVG иконки: `fetch → r.text() → svgCache.set(src, text)`
- Параллельные запросы к тому же URL дедуплицируются через inflight-мапу
- Второй и далее — мгновенное чтение из памяти
- *Обходит HTTP-кэш браузера* (важно: на nginx стоит max-age=0, так что `new Image()` для SVG был бы бесполезен)
2. Asset prewarm
src/lib/characterAssets.ts:131-157 — `prewarmCharacterContent():`
3. Данные — вызывает `loadCharacterData()` (синглтон-промис выше)
4. SVG иконки — `prefetchSvgIcons()` (fire-and-forget, в JS-кэш)
5. Тяжёлые PNG/JPG (фон шагов + hero-изображения персонажей) — preloadImages() через `new Image()`, с прогресс-колбэком и таймаутом 30с
6. Флаг warmed — защита от повторного прогрева
7. Retry-прослойка для API
`src/services/_retry.ts` — экспоненциальный backoff для GET-запросов справочников `(/api/character-builder/*):`
`export const withRetry = async <T>(fn: () => Promise<T>, attempts = 3, baseDelayMs = 400) => ...`
- Паузы: 400ms → 800ms → 1600ms (`baseDelay * 2^i`)
- 4xx не ретраим — это ошибки приложения (403, 404 и т.п.)
- 5xx и сетевые ошибки ретраим до 3 раз
- Используется в `useSpellsForSelection`, `useSubclassesForClass`, `useSubracesForRace`
2. Хук-потребитель
`useCharacterCatalog()` в `hooks/character/useCharacterCatalog.ts:`
- Вызывает `loadCharacterData()` и `prewarmCharacterContent()` параллельно в useEffect
- Показывает `bootstrapping:` true пока данные не пришли и ассеты не прогреты
- Прогресс отображается в `CharacterPreloader` (0–100%)
Схематично
useCharacterCatalog (хук)
	│
	├─ loadCharacterData() ──┐
	│ ▼
	│ dataPromise (синглтон)
	│ │
	│ ├─ getRaces() ── withRetry ── fetch /api/character-builder/races
	│ ├─ getClasses() ── withRetry ── fetch /api/character-builder/classes
	│ └─ ...
	│
	└─ prewarmCharacterContent()
			 │
			 ├─ prefetchSvgIcons() ── fetch → svgCache (Map в памяти)
			 └─ preloadImages() ── new Image() (HTTP-кеш браузера)
Итог
Кэш не персистентный — живёт *только в памяти текущей вкладки*. При перезагрузке страницы все данные *фетчатся заново*. Это осознанное решение: справочники персонажа небольшие (десятки элементов), а сложность инвалидации при изменениях в бэке не оправдывает localStorage/IndexedDB.

### Дослучивание

Механизм, при котором генерация случайного персонажа происходит асинхронно, потому что:
- подкласс можно выбрать только если выбран класс(подкласс *грузится лениво*) и только если он существует для 1 уровня
- подрасу можно выбрать только выбрав расу и только если у данной расы есть подраса
- набор заклинаний зависит от класса
- набор навыков зависит от класса

 **Механизм дослучивания** (строки 103–109, 233–282)
`randomizeCharacter()`
  ├── устанавливает расу, класс и т.д.
  ├── pendingSubRace.current = true   ← флаги useRef
  ├── pendingSubClass.current = true
  └── pendingSpells.current = true
  
       ↓ (после рендера, когда приехали опции)
       
`useEffect` → проверяет `pendingSubRace`
  ├── `subRacesLoading?` → жди
  ├── `subRaceOptions пуст?` → жди (ещё не загрузились)
  ├── `нет raceId?` → сбрось флаг (у расы нет подрас)
  └── pickRandom(subRaceOptions) → adaptSubRaceOption → setSelection
      → pendingSubRace.current = false
      
 **Три useEffect** (строки 239–282) *работают идентично*:
1. Подраса (`pendingSubRace`, строка 239): ждёт `!subRacesLoading && subRaceOptions.length > 0.` Если у расы нет подрас (`!raceId`) — гасит флаг и выходит.
2. Подкласс (`pendingSubClass`, строка 254): симметрично, ждёт опции подкласса для текущего класса.
3. Заклинания (`pendingSpells`, строка 269): ждёт `!spellsLoading`, затем тасует кантрипы(заговоры) и заклинания.
