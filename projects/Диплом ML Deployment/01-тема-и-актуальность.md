# 📋 Конкретная постановка проблемы

## Суть проблемы

**60% ML-проектов не доходят до production** (VentureBeat AI Summit), и даже те, что доходят, сталкиваются с систематическими проблемами:

### Проблема 1: ML ≠ Traditional Software

| Аспект | Traditional Software | ML System |
|--------|---------------------|-----------|
| Behavior | Deterministic code | Probabilistic, data-dependent |
| Testing | Unit tests | Validation metrics |
| Reproducibility | git commit | data + code + hyperparameters |
| Updates | Deploy code | Retrain + redeploy |
| Failure | Crash/error | Silent degradation |

**Проблема:** Existing software engineering practices insufficient for ML.

### Проблема 2: The Deployment Gap

```
┌────────────────────────────────────────────────────────────┐
│  DATA SCIENCE TEAM                    INFRA TEAM            │
│  ┌─────────────┐                    ┌─────────────┐       │
│  │ Jupyter     │ ─── "works on my   │ Production  │       │
│  │ Notebook    │      machine" ───→ │ Kubernetes  │       │
│  └─────────────┘                    └─────────────┘       │
│        ↓                                    ↓              │
│  Python, pandas,                    Java, microservices    │
│  scikit-learn                       CI/CD, monitoring      │
└────────────────────────────────────────────────────────────┘
```

### Проблема 3: Scalability Challenges

| Challenge | Description |
|------------|-------------|
| Computational | Model size: 100MB → 10GB+ |
| Traffic | 10 req/min → 10,000 req/sec |
| Latency | <50ms required in production |
| GPU | Expensive, limited availability |
| Autoscaling | CPU-based HPA wrong for ML |

### Проблема 4: Versioning & Lifecycle

**Model graveyard syndrome:**
```
Model v1 → deployed  →  "Model is broken!"
Model v2 → deployed  →  "v1 was better for Segment C!"
Model v1.1 → deployed →  "Forgot to track changes"
```

### Проблема 5: Silent Failures

```
Traditional: Request → Error → Alert → Human reacts
ML System:    Request → "42%" → Looks OK → Model silently broke
```

**Example:** Model trained on 2020 data predicting 2024 trends — predictions valid, but wrong.

### Проблема 6: Team Silos

| Stage | Issue |
|--------|-------|
| Training → Deployment | "Works in notebook" |
| Deployment → Monitoring | "No visibility into model" |
| Monitoring → Retraining | "When do we retrain?" |

### Проблема 7: Security

| Threat | Impact |
|--------|--------|
| Model inversion | Reconstruct training data |
| Adversarial attacks | Fool the model |
| Model theft | Copy via API queries |
| Data poisoning | Compromised model |

---

## Почему существующие решения недостаточны

| Решение | Limitations |
|---------|-------------|
| Jupyter in production | Not designed for serving |
| AWS SageMaker | Vendor lock-in, expensive |
| Kubeflow | Steep learning curve, Kubernetes-native only |
| MLflow | Primarily tracking, not deployment |
| Seldon | Limited Java ecosystem integration |

**GAP:** No open-source, language-agnostic, Java-friendly framework for ML deployment in microservices.

---

## Как диплом решает эти проблемы

| Проблема | Решение | Компонент |
|----------|---------|-----------|
| ML ≠ Software | Microservice separation | Model Service, Business Service |
| Deployment gap | Full pipeline | Model Registry + CI/CD |
| Scaling | K8s HPA + GPU | Inference Service with autoscaling |
| Versioning | Model Registry | Version control for models |
| Silent failures | ML Observability | Prometheus + Grafana dashboards |
| Team silos | API contracts | gRPC/REST interfaces |
| Security | Canary deploys | Gradual rollout + rollback |
