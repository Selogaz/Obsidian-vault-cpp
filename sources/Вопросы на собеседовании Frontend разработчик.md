# Event Loop

Event Loop — это фундаментальный механизм, который позволяет однопоточному JavaScript выполнять неблокирующие операции: таймеры, сетевые запросы, чтение файлов. Сейчас я подробно разложу, из чего он состоит и в каком порядке всё выполняется.

1. Базовые составляющие

Представьте себе четыре главных компонента:

· Call Stack (стек вызовов) — синхронный исполнитель. Функция попала в стек — выполнилась до конца, стек опустел. Пока в стеке есть хоть что-то, браузер «заморожен».
· Web APIs / libuv — сторонние механизмы, не часть движка JS. Здесь живут таймеры (setTimeout), сетевые запросы (fetch/XHR), DOM-события, чтение файлов (в Node.js) и т.д.
· Очереди задач (Task Queues) — буферы, куда попадают колбэки после завершения асинхронной операции. Их две:
	· Macrotask очередь (Task Queue) — setTimeout, setInterval, I/O, события UI, setImmediate (Node).
	· Microtask очередь — Promise.then/catch/finally, queueMicrotask, MutationObserver.
· Сам Event Loop — бесконечный цикл, который постоянно смотрит на стек и очереди.

1. Алгоритм одной итерации Event Loop (по шагам)

Цикл живёт, пока есть хотя бы одна задача. Грубо говоря, он делает так:

1. Взять один макротаск из очереди макрозадач (если очередь пуста — ждать). Положить его в Call Stack.
2. Выполнить этот макротаск (стек опустеет).
3. Как только стек пуст, начинается микротаск-чекаут (microtask checkpoint):
	 · Забираем все микротаски из очереди и выполняем их одну за другой, пока очередь микротасков не опустеет. Если во время выполнения микротаска порождает новый микротаск, он попадёт в ту же очередь и будет выполнен в этой же фазе — стек не успеет освободиться для макротасков.
4. После того как очередь микротасков полностью опустела, возможен рендеринг (в браузере), если он запланирован.
5. Вернуться к шагу 1 — взять следующий макротаск.

Главное правило: один макротаск → все микротаски → следующий макротаск.

1. Пример порядка выполнения

Возьмём такой код и разберём, что выведется в консоль:

console.log('1');

setTimeout(() => console.log('2'), 0);

Promise.resolve().then(() => console.log('3'));

console.log('4');
Порядок:

· Сначала синхронный код: в стек попадают console.log('1') → выводит 1; планируется setTimeout (его колбэк уходит в Web API и сразу же попадает в macrotask очередь); регистрируется Promise.then (колбэк попадает в microtask очередь); console.log('4') → выводит 4. Стек пуст.
· Микротаск-чекаут: очередь микротасков — () => console.log('3'). Выполняется → выводит 3. Очередь микротасков пуста.
· Event Loop берёт следующий макротаск: колбэк setTimeout → выводит 2.

Итог: 1, 4, 3, 2.

Это объясняет, почему setTimeout(..., 0) не выполняется мгновенно — он макротаск, и микротаски обгоняют его всегда.

1. Микротаски против макротасков: ключевые различия

· Микротаски — высокоприоритетные, выполняются сразу после текущего стека и до следующего макротаска. Даже если макротаск уже давно ждёт, сначала опустошится очередь микротасков.
· Макротаски — следующая «тяжёлая» работа: рендеринг браузера, пользовательские события, таймеры.
· Если в микротаске бесконечно порождать новые микротаски (например, Promise.resolve().then(function recursive() { /* снова then */ })), очередь никогда не опустеет, и макротаски (включая рендеринг) никогда не выполнятся — страница зависнет.

1. Роль async/await

await — это синтаксический сахар над Promise. Всё, что идёт после await, попадает в микротаск-очередь, как .then.

async function example() {
	console.log('A');
	await null; // эквивалентно Promise.resolve(null).then(() => ...)
	console.log('B');
}
example();
console.log('C');
Вывод: A, C, B. Потому что после await продолжение функции становится микротаском.

1. Рендеринг и requestAnimationFrame (браузер)

Между макротасками, после очистки микротасков, браузер может выполнять рендеринг (если нужно). Колбэки requestAnimationFrame — это ещё одна отдельная очередь, которая обрабатывается прямо перед рендерингом. То есть:

· Макротаск → все микротаски → очередь requestAnimationFrame (если есть) → перерисовка → следующий макротаск.

1. Node.js особенности

В Node.js Event Loop имеет несколько фаз (timers, pending callbacks, poll, check, close callbacks), и каждая фаза обрабатывает свою очередь макротасков. Но правило с микротасками работает так же: между каждой фазой и после каждого макротаска выполняются все накопившиеся микротаски (и process.nextTick, у которого ещё более высокий приоритет в Node). Однако на собеседовании по фронтенду обычно достаточно браузерной модели.

1. Распространённые уточняющие вопросы и короткие ответы на них

— Почему setTimeout не гарантирует точную задержку?
Потому что колбэк будет вызван не раньше, чем через указанное время, но реально — когда освободится стек, закончатся текущие макротаски и все микротаски перед ним.

— Что произойдет, если поместить setTimeout внутрь микротаска?
Он будет зарегистрирован и его колбэк попадёт в очередь макротасков. Сам микротаск закончится, затем (если больше нет микротасков) Event Loop перейдёт к макротаскам.

— Как не блокировать рендеринг?
Разбивать долгие вычисления на части, используя setTimeout (передавать управление циклу) или requestAnimationFrame для визуальных задач. Так между кусками вычислений Event Loop сможет выполнять рендеринг.

— В чём разница между setTimeout(f, 0) и queueMicrotask(f)?
queueMicrotask выполнит переданную функцию в текущем микротаск-чекауте, до любого макротаска (и до рендеринга). setTimeout — это новый макротаск, он будет ждать своей очереди после микротасков.

Заключение

Event Loop — это элегантное решение, которое при одном потоке обеспечивает асинхронность, реактивность и отсутствие гонок за счёт чёткого порядка: стек → все микротаски → один макротаск → снова микротаски. Понимание этого порядка критически важно для предсказуемой работы с асинхронным кодом, таймерами и промисами.
