[[java middle Verme легенда]]

Тип проекта и команда
— Что за продукт? Состав команды?
 Проект - веб-платформа Verme, предназначенная для повышения эффективности управления линейным персоналом: учёт рабочего времени, автопланирование смен, биометрическая идентификация, инструменты для работы с внешним персоналом. Команда: 1 техлид, 2 бэкенда, 2 фронтенд-разработчика, 2 QA-инженера, аналитик, формализующий требования и составляющий ТЗ. Работали по двухнедельным спринтам в рамках Agile-подхода.

Твои зоны ответственности
— За какие модули/сервисы отвечал?
Реализовывал и поддерживал ключевые модули, связанные с управлением сменами и планированием графиков - создание, редактирование смен, автоматическое планирование с использованием асинхронных очередей. Отвечал за разработку и интеграцию REST API для взаимодействия с внешними системами.

— Какие задачи решал регулярно?
- Интегрировал backend-сервисы с фронтендом через REST API и с базой данных
- Поддерживал корректную реализацию бизнес-логики, обрабатывал ошибки и крайние случаи
- Писал юнит- и интеграционные тесты для покрытия критически важных сценариев
- Составлял техническую документацию по новому функционалу и описывал API в Swagger

Ключевые задачи
— Какие фичи/модули лично реализовал?
- Внедрил систему автоматизированного планирования графиков с использованием асинхронных очередей и Redis
- Оптимизировал REST API с использованием Spring MVC, который используется фронтендом и внешними сервисами

— Что оптимизировал/рефакторил?
Оптимизировал структуру базы данных для хранения данных о сменах сотрудников, что ускорило время выборки данных. Перевел ключевые эндпоинты API на асинхронную обработку запросов. Рефакторил код модулей планирования, выделяя общие части в отдельные сервисы.

— С какими трудностями сталкивался и как решал?
Впервые работал с асинхронными очередями и Redis - изучал документацию, разбирался с их особенностями, настраивал обработку фоновых задач. Осваивал паттерны асинхронной обработки для обеспечения надежности системы.

- Расскажи о самой интересной/самой сложной задаче (самое важное из этого блока)
Кейс 1. Внедрение системы асинхронного планирования графиков
Проблема: процесс автоматического планирования смен для сотен сотрудников выполнялся синхронно и занимал несколько секунд, что приводило к таймаутам запросов и плохому UX.
Решение: спроектировал и внедрил систему на основе Redis и асинхронных очередей. Разбил процесс планирования на отдельные фоновые задачи. Реализовал обработку ошибок и ретраи для надежности.
Результат: время обработки запросов на планирование сократилось на 20%, система стала более устойчивой к нагрузкам

Кейс 2: Оптимизация REST API для интеграции с внешними платформами
Проблема: API для интеграции с внешними системами и мобильными приложениями не справлялось с растущей нагрузкой, особенно при массовых операциях с данными сотрудников.
Решение: провел анализ узких мест в производительности, идентифицировал наиболее нагруженные эндпоинты. Перевел критические операции на асинхронную обработку с использованием CompletableFuture и @Async. Реализовал пагинацию для эндпоинтов возвращающих большие объемы данных и добавил кэширование часто запрашиваемых данных.
Результат: производительность API увеличилась на 35%, время отклика для массовых операций сократилось на 4 секунды, система стабильно работает при одновременной работе 1000+ пользователей.

Технологии и инструменты
— Что из стека использовал ?
Java 17, Spring Boot, Hibernate, PostgreSQL, Spring Security, REST API, Spring MVC, Maven, JUnit, Mockito, Docker, Git.

— Какие инструменты автоматизации/CI/CD?
Jenkins выполнял задачи по сборке проекта, запуску unit и интеграционных тестов, а также сборке Docker-образов. GitLab CI/CD использовался для автоматизации деплоя на тестовые и продакшн окружения

Тесты и качество
— Как тестировал код?
Писал unit-тесты для бизнес-логики и сервисных слоев, создавая заглушки и имитации зависимостей. Интеграционные тесты запускал с использованием Spring Boot Test, проверяя взаимодействие компонентов и корректность работы с базой данных PostgreSQL в тестовом окружении.

— Делал ли ревью чужого кода?

Результаты (здесь важно добавить цифры)
— Что улучшилось благодаря твоим решениям? (метрики, скорость, стабильность)
- Внедрил систему асинхронного планирования с Redis, что сократило время обработки запросов на 20%

Решил сменить проект, чтобы получить более разнообразный и глубокий опыт в backend-разработке, повысить свою ценность как разработчика.
