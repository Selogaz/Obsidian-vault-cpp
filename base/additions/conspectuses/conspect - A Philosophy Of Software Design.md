# Preface

Главная проблема в computer science - как взять большую проблему и разделить её на части, которые могут быть *решены независимо*.
В то же время, **нет ни одной аудитории в университетах, где бы эта проблема была в центре внимания** - вместо этого учат синтаксису и [[ООП]]

Существует много научных доказательств, что выдающиеся результаты связаны в большей степени с *высококачественной практикой*, чем с врожденным талантом.
Процесс, из которого родилась книга - студенты писали код, делали ошибки и видели, как их ошибки и последующие исправления относятся к принципам проектирования

Идеи автора пришли из личного опыта написания и чтения кода. В своей карьере он написал около 250к строк кода в разных языках. Принимал участия в самых разных проектах, в которых на собственном опыте ощутил проблемы больших систем. Прочитал много кода, написанного другими людьми.

Главная цель книги - уменьшение сложности приложений.

# Introduction

Главное ограничение в написании ПО - способность понимать системы, которые мы создаем.
Сложность программ *неизбежно* возрастает с течением времени.

Сложность систем **ограничивает вещи, которые мы можем делать в одиночку**

Сложность может быть уменьшена с помощью приведения *особых случаев к общему* виду и использования *единого стиля.*
Модульность - *способ не испытывать трудности* от сложности сразу всей программы, контактируя лишь с отдельными её частями по отдельности.

## Подходы в разработке

### Waterfall Model

Waterfall model - подход, в котором проект разделен на отдельные фазы(requirements definition, design, coding, testing, and maintenance).
В нем каждая фаза заканчивается там, где начинается другая. Как правило, за каждую фазу отвечают разные люди.

**Плохо работает** для программного обеспечения потому что *невозможно визуализировать* дизайн большой программной системы достаточно хорошо *до её реализации* - изначальный дизайн будет иметь множество проблем. Проблемы **не становится очевидными** до тех пор, пока не начнется реализация.

При таком подходе, разработчикам приходится *обходить проблемы, не меняя дизайн* и это приводит к **огромному росту сложности**.

### [[java agile|agile]]

В отличие от waterfall, фокусирует начальный дизайн на *небольших частях* общей функциональности. Вследствие чего, *недостатки выявляются и исправляются раньше*. Более поздние фичи получаются **качественнее**, потому что впитывают в себя *опыт от исправления ошибок предыдущих фич*

Данный инкрементальный(последовательный?каким словом заменить?) подход работает, потому что программы достаточно гибкие для того, чтобы *выдержать значительные изменения* на этапе реализации. Чего не скажешь о физических системах.

Инкрементальный подход подразумевает, что разработка ПО никогда не заканчивается. Проблемы находятся и решаются, дизайн переделывается etc.

Так происходит потому что начальный дизайн **никогда не является самым лучшим**; опыт неизбежно показывает пути сделать ПО лучше.
Поэтому, следует *планировать время для дизайнерских[^1] улучшений[^2]*.

### Afterwords

Эта книга ответит на вопрос: "Как использовать сложность для управления разработкой ПО на протяжении всего его жизненного цикла?". У книги 2 цели:
- Описать природу сложности ПО
- Дать техники для уменьшения её сложности

## How to use this book

Читать чужой код и думать о том, как этот код связан с концепциями книги и сложностью кода. *Ревью чужого кода познакомит с новыми подходами и программистскими техниками*

Чем больше разных подходов будет использовано в процессе решения проблемы, тем больше будет выучено

Каждое правило имеет исключения, каждый принцип имеет пределы.
