# Проблема: ML Deployment Gap

## Центральный исследовательский вопрос

> **Как эффективно развернуть ML-модели (нейросети) в production-среде, обеспечив масштабируемость, наблюдаемость и maintainability?**

## Sub-problems

### Sub-problem 1: Computational scaling

**Question:** How to scale ML inference horizontally under variable load?

**Current state:** CPU-based autoscaling insufficient for GPU-bound workloads.

**Approach:** Custom metrics-based HPA (queue depth, GPU utilization).

---

### Sub-problem 2: Model lifecycle management

**Question:** How to version, track, and roll back ML models as reliably as software?

**Current state:** Models stored in S3 with ad-hoc naming.

**Approach:** Model Registry with metadata, lineage, and rollback support.

---

### Sub-problem 3: ML-specific monitoring

**Question:** How to detect when a model degrades in production?

**Current state:** Traditional APM (CPU, memory) misses ML-specific failures.

**Current state:** Traditional APM (CPU, memory) misses ML-specific failures.

**Approach:** Data drift detection, prediction distribution monitoring.

---

## Scope

**In scope:**
- Microservices architecture for ML deployment
- Model serving (not training pipeline)
- Monitoring and observability
- Autoscaling

**Out of scope:**
- ML model training
- Feature engineering
- Model optimization (quantization, pruning)
- Distributed training

**Rationale:** Focus on infrastructure/DevOps aspect (matches user's skills), while ML-component is integration (not development).
