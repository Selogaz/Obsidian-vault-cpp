================================================================
0 Введение и цель документа
================================================================

Данный документ описывает подробное техническое задание на разработку
frontend-части онлайн-платформы DnD с ИИ-мастером.

Цель документа — зафиксировать ВСЕ пользовательские экраны, действия,
состояния и сценарии, которые будут реализованы во frontend.

================================================================
0.1 Общие принципы и ограничения
================================================================

0.1.1 Роль Frontend
Frontend отвечает исключительно за:
- отображение пользовательского интерфейса
- пользовательские сценарии
- визуальные состояния
- блокировки/разблокировки элементов интерфейса

Frontend НЕ:
- рассчитывает игровые механики
- принимает решения по бою, экономике или правилам
- хранит постоянные игровые состояния

0.1.2 Источник истины
- Backend является единственным источником истины
- Любые данные, приходящие от backend, считаются приоритетными
- Frontend обязан синхронизировать UI с backend в реальном времени

0.1.3 Общие UX-принципы
- Любое недоступное действие визуально заблокировано
- Пользователь всегда понимает, почему действие недоступно
- Все ошибки отображаются текстом, понятным пользователю


================================================================
1 Общая архитектура Frontend
================================================================

Frontend реализуется как SPA-приложение на Vue.js и выполняет роль:
- пользовательского интерфейса платформы
- клиента для REST API
- клиента для WebSocket-соединений
- слоя интеграции Unity WebGL
- слоя пользовательского ввода (клики, текст, голос)

Frontend не содержит игровой логики и не принимает решений,
все ключевые состояния и правила поступают с backend.

================================================================
1.1 Используемый стек
================================================================

- Vue.js 3
- Vite (сборка проекта)
- Composition API
- Pinia (state management)
- TailwindCSS + кастомные UI-компоненты
- WebSocket (реалтайм-обновления)
- WebRTC (голосовое общение между игроками)
- Unity WebGL (игровая карта и визуализация)

================================================================
1.2 Структура проекта
================================================================

Рекомендуемая структура frontend-проекта:

/src
 ├── assets/          (изображения, иконки, шрифты)
 ├── components/      (переиспользуемые UI-компоненты)
 ├── views/           (страницы, связанные с роутами)
 ├── modules/         (крупные функциональные модули)
 │    ├── auth/
 │    ├── profile/
 │    ├── characters/
 │    ├── lobby/
 │    ├── game/
 │    ├── shop/
 │    └── support/
 ├── store/           (Pinia-хранилища)
 ├── services/        (REST и WebSocket клиенты)
 ├── unity/           (Unity WebGL bridge)
 ├── utils/           (утилиты и хелперы)
 └── types/           (типы и интерфейсы)

Структура позволяет масштабировать проект без рефакторинга ядра.

================================================================
1.3 WebSocket-клиент
================================================================

Frontend использует единый WebSocket-менеджер, который:
- инициализируется при входе в лобби или игру
- привязывается к конкретной комнате
- автоматически переподключается при обрыве соединения

Типы обрабатываемых событий:
- lobby_update
- room_status
- game_state
- combat_state
- chat_message
- system_message
- dice_roll
- movement_update
- economy_update
- pause_state

Все события:
- обрабатываются централизованно
- обновляют Pinia-хранилища
- приводят к обновлению UI

================================================================
1.4 Управление состояниями (State Management)
================================================================

Основные хранилища:

- userStore
  - данные пользователя
  - авторизация
  - баланс

- characterStore
  - список персонажей
  - выбранный персонаж

- roomStore
  - состояние лобби
  - список игроков
  - статус комнаты

- gameStore
  - состояние игры
  - текущий ход
  - экономические состояния
  - паузы

- chatStore
  - чат
  - боевой лог
  - системные сообщения

- wsStore
  - статус соединения
  - состояние переподключения

================================================================
1.5 Интеграция Unity WebGL
================================================================

Unity WebGL загружается ТОЛЬКО при старте игровой сессии.

Frontend:
- инициализирует Unity через loader
- отображает индикатор загрузки
- передаёт Unity стартовые данные

Unity НЕ загружается:
- на главной странице
- в личном кабинете
- в магазине
- в лобби

Обмен данными:
Frontend → Unity:
- команды перемещения
- игровые события
- идентификаторы комнаты и персонажа

Unity → Frontend:
- подтверждения действий
- события боя
- анимации кубиков
- обновления карты

2. Глобальные UI-состояния
================================================================

2.1 Загрузка
- Глобальный loader отображается:
  - при первом входе пользователя
  - при переходе между ключевыми экранами
  - при ожидании критичных данных (игра, лобби, магазин)
- Во время загрузки:
  - все кнопки и инпуты заблокированы

2.2 Ошибки
- Ошибки приходят от backend в текстовом виде
- Ошибки могут быть:
  - глобальными (модальное окно)
  - локальными (под конкретным элементом)
- Пользователь всегда видит причину ошибки

2.3 Потеря соединения
- Отображается полноэкранное сообщение
- Все действия блокируются
- После восстановления соединения:
  - происходит синхронизация состояния
  - пользователь возвращается в актуальный экран

6. Создание персонажа 
================================================================

Создание персонажа реализуется в виде пошагового мастера.
Переход между шагами линейный, пропуск шагов невозможен.

Шаги мастера:

6.1 Имя персонажа
- Ввод имени
- Проверка длины и допустимых символов

6.2 Портрет персонажа
- Выбор портрета из доступных
- Предпросмотр

6.3 Пол персонажа
- Выбор M / F
- После подтверждения изменить нельзя

6.4 Раса и подраса
- Выбор расы
- При необходимости выбор подрасы
- Отображение краткого описания

6.5 Класс и подкласс
- Выбор класса
- Выбор подкласса
- Отображение базовой информации

6.6 Происхождение
- Выбор бэкграунда персонажа
- Краткое описание

6.7 Характеристики
- Распределение очков (Point Buy)
- Отображение доступных очков
- Запрет выхода за лимиты

6.8 Стартовая экипировка
- Отображение стартового набора
- Только просмотр (без выбора)

6.9 Подтверждение
- Сводка всех выбранных параметров
- Кнопка «Создать персонажа»

8. Лобби (pre-game)
================================================================

8.1 Назначение
- Сбор игроков
- Проверки перед стартом игры
- Игра может быть начата одним игроком

8.2 Элементы интерфейса
- Заголовок «Лобби»
- Список игроков
- Миниатюра сценария
- Кнопка «Пригласить друга»
- Кнопка «Начать» (только хост)
- Индикатор готовности
- Информационный текст об оплате
- Бегущая строка подсказок

8.3 Поведение
- Unity не загружается
- Голосовой чат доступен
- Команды ИИ недоступны

================================================================
9. Игровой стол
================================================================

9.1 Общий layout
- Верхняя панель: статус, экономика
- Левая панель: действия и очередь ходов
- Центральная область: Unity WebGL
- Правая панель: чат-лог
- Нижняя панель: игроки
- Карточка персонажа с заметками

9.2 Действия игрока
- Бросить кубик
- Ввод текстовой команды
- Голосовая команда ИИ (только в свой ход)

9.3 Ограничения
- Вне своего хода действия заблокированы
- При экономической паузе действия заблокированы

9.4 Рассинхронизации состояния
- UI временно блокируется
- отображается сообщение «Синхронизация состояния»
- после подтверждения backend UI разблокируется

14. Страница «Новости»
================================================================

14.1 Назначение страницы
Страница «Новости» предназначена для:
- информирования пользователей о новостях проекта
- публикации обновлений, анонсов, патчей
- повышения вовлечённости пользователей
- привлечения подписчиков в соц сетях
- Усиления SEO ядра

14.2 Структура страницы
- Заголовок страницы «Новости»
- Список новостных карточек
- Пагинация или lazy-load (определяется backend)

14.3 Карточка новости
Каждая карточка новости содержит:
- Обложку (изображение)
- Заголовок новости
- Краткое описание (анонс)
- Дату публикации
- Переход на пост в соц.сети

14.4 Поведение карточки
- Клик по карточке открывает страницу новости
- Клик по заголовку эквивалентен клику по карточке
- Клик по кнопке соц сети переносит пользователя на пост в соц сети

14.5 Страница отдельной новости
Содержит:
- Заголовок
- Основное изображение
- Тело новости (текст + изображения)
- Возможность встраивания видео
- Дату публикации

14.6 Ограничения
- Контент только текст и видео
- Комментарии отсутствуют
- Лайки / реакции отсутствуют

================================================================
15. Страница «Гайды»
================================================================

15.1 Назначение страницы
Страница «Гайды» предназначена для:
- обучения новых игроков
- объяснения механик платформы
- снижения порога входа

15.2 Структура страницы
- Заголовок «Гайды»
- Список гайдов в виде карточек
- Возможность фильтрации (если предусмотрено backend)

15.3 Карточка гайда
Каждая карточка гайда содержит:
- Иконку или изображение
- Название гайда
- Краткое описание
- Примерную длительность изучения

15.4 Поведение
- Клик по карточке открывает страницу гайда

15.5 Страница гайда
Содержит:
- Заголовок
- Контент гайда:
  - текст
  - встроенное видео
- Контент структурирован (подзаголовки, абзацы)

15.6 Ограничения
- Контент только текст и видео
- Нет комментариев
- Нет оценок

16. Страница «Создание игры»
================================================================

16.1 Назначение страницы
Страница «Создание игры» предназначена для:
- выбора сценария (истории)
- настройки параметров будущей игры
- создания игровой комнаты

================================================================
16.2 Блок «Истории»
================================================================

16.2.1 Список историй
Истории отображаются в виде карточек.

Каждая карточка истории содержит:
- Изображение (обложка)
- Название истории
- Краткое описание
- Рекомендуемое количество игроков
- Примерное время игры

16.2.2 Кнопка «Выбрать»
- Под каждой историей расположена кнопка «Выбрать»
- При нажатии:
  - выбранная история подсвечивается зелёной рамкой
  - предыдущий выбор (если был) снимается

================================================================
16.3 Фильтры историй
================================================================

Фильтры отображаются в отдельном блоке.

16.3.1 Фильтр по жанру
Доступные жанры:
- Фэнтези
- Ужасы
- Приключения
- Мистика
- Детектив
- Комедия

Возможен выбор одного или нескольких жанров.

16.3.2 Фильтр по количеству игроков
Варианты:
- 1–2
- 3–4
- 4–6
- 6–8
- 8–10

16.3.3 Фильтр по времени игры
Варианты:
- 1–4 часа
- 4–12 часов
- 12–26 часов
- 26–40 часов
- 40–80 часов
- от 80 часов

16.3.4 Фильтр сложности
- Лёгкая
- Средняя
- Сложная

16.3.5 Поведение фильтров
- Фильтры применяются без перезагрузки страницы
- Возможна комбинация нескольких фильтров
- При отсутствии результатов отображается сообщение

================================================================
16.4 Создание игры
================================================================

16.4.1 Проверки
Перед созданием игры ОБЯЗАТЕЛЬНО:
- выбрана история
- выбрана сложность

Если условия не выполнены:
- кнопка «Создать игру» неактивна

16.4.2 Кнопка «Создать игру»
- Активируется только при корректных данных
- При нажатии отправляется запрос на backend

================================================================
16.5 Успешное создание игры
================================================================

16.5.1 Модальное окно
После успешного создания игры отображается модальное окно с:
- Номером комнаты
- Паролем комнаты

16.5.2 Кнопки модального окна
- «Скопировать»:
  - копирует номер и пароль комнаты
- «Продолжить»:
  - переводит пользователя на страницу создания персонажа

16.5.3 Поведение
- Модальное окно блокирует взаимодействие с фоном
- Закрытие возможно только через предусмотренные кнопки

================================================================
17. Навигационный сценарий после создания игры
================================================================

Последовательность:
Создание игры
→ Успешное модальное окно
→ Кнопка «Продолжить»
→ Экран создания персонажа
→ Экран выбора персонажа
→ Лобби

18 Производительность и устойчивость
================================================================

Требования:
- рендер Unity не должен блокировать интерфейс;
- реакция на WebSocket ≤ 150 мс;
- чат до 10 000 сообщений без лагов;
- FPS Unity ≥ 30 на средних ПК
