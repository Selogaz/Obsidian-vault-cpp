**кэш подземелья максвелла**
—
Проект использует трёхуровневую систему кэширования для данных и ассетов визарда персонажа.

1. **Модульный синглтон-промис (данные)**
   `src/lib/characterAssets.ts:94-114` — самый верхний уровень. Промис загрузки справочных данных (расы, классы, происхождения, мировоззрения) сохраняется в замыкании:
   ```ts
   let dataPromise: Promise<CharacterBootstrapData> | null = null;

   export function loadCharacterData(): Promise<CharacterBootstrapData> {
     if (!dataPromise) {
       dataPromise = Promise.all([getRaces(), getClasses(), getBackgrounds(), getAlignments()])
         .then(([races, classes, backgrounds, alignments]) => ({ races, classes, backgrounds, alignments }));
     }
     return dataPromise;
   }
   ```
   Первый вызов фетчит с бэка, все последующие — возвращают тот же промис. Бессрочный кэш на время жизни страницы (сброс только через `resetCharacterCache()`).

2. **In-memory SVG-кэш**
   `src/lib/svgCache.ts` — простая `Map<string, string>`:
   ```ts
   const svgCache = new Map<string, string>();
   const inflight = new Map<string, Promise<void>>();
   ```
   - При первом запросе SVG иконки: `fetch → r.text() → svgCache.set(src, text)`
   - Параллельные запросы к тому же URL дедуплицируются через inflight-мапу
   - Второй и далее — мгновенное чтение из памяти
   - *Обходит HTTP-кэш браузера* (на nginx стоит max-age=0, так что `new Image()` для SVG был бы бесполезен)

3. **Asset prewarm**
   `src/lib/characterAssets.ts:131-157` — `prewarmCharacterContent()`:
   - Данные — вызывает `loadCharacterData()` (синглтон-промис выше)
   - SVG иконки — `prefetchSvgIcons()` (fire-and-forget, в JS-кэш)
   - Тяжёлые PNG/JPG (фон шагов + hero-изображения персонажей) — `preloadImages()` через `new Image()`, с прогресс-колбэком и таймаутом 30с
   - Флаг warmed — защита от повторного прогрева

4. **Retry-прослойка для API**
   `src/services/_retry.ts` — экспоненциальный backoff для GET-запросов справочников (`/api/character-builder/*`):
   ```ts
   export const withRetry = async <T>(fn: () => Promise<T>, attempts = 3, baseDelayMs = 400) => ...
   ```
   - Паузы: 400ms → 800ms → 1600ms (`baseDelay * 2^i`)
   - 4xx не ретраим — ошибки приложения (403, 404)
   - 5xx и сетевые ошибки ретраим до 3 раз
   - Используется в `useSpellsForSelection`, `useSubclassesForClass`, `useSubracesForRace`

5. **Хук-потребитель**
   `useCharacterCatalog()` в `hooks/character/useCharacterCatalog.ts`:
   - Вызывает `loadCharacterData()` и `prewarmCharacterContent()` параллельно в useEffect
   - Показывает `bootstrapping: true` пока данные не пришли и ассеты не прогреты
   - Прогресс отображается в `CharacterPreloader` (0–100%)

## Схематично

```
useCharacterCatalog (хук)
  │
  ├─ loadCharacterData() ──┐
  │ ▼                       │
  │ dataPromise (синглтон)   │
  │ │                       │
  │ ├─ getRaces() ── withRetry ── fetch /api/character-builder/races
  │ ├─ getClasses() ── withRetry ── fetch /api/character-builder/classes
  │ └─ ...                   │
  │                          │
  └─ prewarmCharacterContent()
       │
       ├─ prefetchSvgIcons() ── fetch → svgCache (Map в памяти)
       └─ preloadImages() ── new Image() (HTTP-кеш браузера)
```

## Итог

Кэш **не персистентный** — живёт только в памяти текущей вкладки. При перезагрузке страницы все данные фетчатся заново. Это осознанное решение: справочники персонажа небольшие (десятки элементов), а сложность инвалидации при изменениях в бэке не оправдывает localStorage/IndexedDB.
