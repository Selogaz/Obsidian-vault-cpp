## ⚠️ CRITICAL BUG FOUND (session 2, later) — KeyManager is broken

Live Zotero API check revealed the first JS run FAILED silently:
`Zotero.BetterBibTeX.KeyManager.find({citationKey})` returns the SAME item
(itemID=1, **ВНУТРЯНКА АЙТИ**) for EVERY citationKey. So:
- `.assign_collections.js` (v1) put ВНУТРЯНКА АЙТИ into all 18 leaf collections →
  "Updated 18; 0 not found" was bogus. Distinct items in emoji collections = **1**.
- `.enrich_authors.js` (v1) inspected ВНУТРЯНКА АЙТИ 34× (it has a real author) →
  "skipped 34" was bogus; the 34 target authors are STILL `Unknown` (verified via API).
- `.unify_tags.js` does NOT use KeyManager (iterates items directly) → genuinely worked (94 items). ✅

**Fix:** match by stable Zotero **itemKey**, not citationKey. itemKeys resolved in
Python via the live read-only API: each orphan.bib title → the single item with that
title inside the `orphan` collection (verified 1:1 for all 98, 0 ambiguous/miss).
- `.assign_by_itemkey.py` → regenerates `.assign_collections.js` (98 itemKeys).
- `.enrich_authors.py` → now emits itemKey-based `.enrich_authors.js` (34).
Both JS now use `Zotero.Items.getByLibraryAndKeyAsync(libraryID, itemKey)`.
